DataReachable DevOps Specifications

This documentation sets out the best practices that should be followed when conducting DevOps at DataReachable. The purpose of this documentation is preventing ad hoc selection on tool chains, project folder layouts, configurations, etc. When setting up rules and conventions, future requirements and flexibility should be taken into consideration, instead of meeting temporary needs. However, evolution is essential for this documentation, subject to the advancement in the DevOps industry. Modifications and refactorizations should happen incrementally.

NOTE: DevOps is still in its infancy, so this documentation is an initial attempt at least for now.

Main Tech Stack

  • Architecture: Microservices
  • Containerization: Docker
  • Auxiliary: CI/CD Tools (Github Action)

Main Deployment Tool Chain

  • Cloud Computing Environment: AWS
  • Container Orchestration: Kubernetes
  • Infrastructure as Code: Terraform and Terragrunt

Although the main tech stack adopts Docker-based containerizaton, for flexibility and future extension, we do not rule out conventional deployment options. Under such consideration, a uniform IaC tool is selected (Terraform). It is indeed an overkill for only Kubernetes, but we do not want to limit our future architectural choices.

Terragrunt is adopted to solve several Terraform issues with modularization.

Workflows

We set out two workflows from development to deployment for application and infrastructure code respectively:

  • Application code workflow
  • Infrastructure code workflow

Important:

Once Terraform and Terragrunt are adpoted, one should ONLY use them to manipulate real-world infrastructures. Other approaches are forbidden, e.g., manual operations through AWS console, to avoid inconsistency.

Workflow Criteria for Application and Infrastructure Code

Please comply with the criteria set out in the following table:

Application Code Infrastructure Code
Version Control System
  • git clone
  • One Repo per app
  • Use branches freely
  • git clone
  • Two Repos per project
    (live and modules repos)
  • Don't use branches
    (Deploy only on master branch)
Local Development
  • Local dependencies and environment
  • Run on localhost
  • Sandbox AWS Cloud Environment
  • terraform apply
  • Terratest
Local Code Changes
  • Change the code
  • Run on localhost
  • Local testing
  • Change the code
  • terraform apply
  • Terratest
  • Use test stages
Code Review Submission
  • Submit a pull request
  • Enforce coding guidelines
  • Submit a pull request
  • Enforce coding guidelines
Automated Tests
  • Use the same CI server
  • Static analysis
  • Unit tests
  • Integration tests
  • End-to-end tests
  • Use the same CI server
  • Static analysis
  • Unit tests
  • Integration tests
  • End-to-end tests
  • terraform plan
Release
  • Documentation
  • git tag
  • Create versioned repo releases
  • Documentation
  • git tag
  • Create versioned repo releases
Deployment
  • Create versioned, immutable artifact
    (Docker images)
  • Documentation
  • Hand to DevOps team
  • Deploy with Terraform and other tools
    (e.g. bash scripts)
  • Limited deployment strategies
  • Handle deployment errors
  • Run deployment on the same CD server
  • Promote immutable, versioned artifacts across environments
    (e.g. Dev to Stage to Prod)

Note:

  1. Some steps of the above table can be omitted according to the scale of DevOps team.
  2. In reality, the Don't use branches rule for the infrastructure code repo is only an ideal situation and can be ignored. Refer to details for details.

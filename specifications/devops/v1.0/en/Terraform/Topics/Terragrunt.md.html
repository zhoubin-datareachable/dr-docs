Terragrunt Best Practices

Terragrunt (TG) is enforced alongside Terrafrom (TF).

Reasons on Enforcing Terragrunt

In the live repo, we always hope to deploy the infrastructure as a whole. However, modularization brings about two major issues:

  • The first issue is module dependency in Terraform, which has to be solved by ad hoc scripting. Otherwise, one has to manually run terraform command in each folder according to application-specific order.
  • The second issue is inputs and outputs connections. A module may need other modules' outputs as inputs, which has to be solved by the terraform_remote_state data source.

The solution is enforcing the usage of Terragrunt which is a wrapper over Terraform, to avoid using ad hoc glue scripts and tricks to connect modules.

Best Practices

1. Terragrunt-Generated File Naming Conventions

To distinguish TG-generated TF files from manually coded TF files, name the former with a prefix "_". These files should be excluded from the git repo (add them in .gitignore). For example,

generate "provider" {
  path = "_provider.tf"
  if_exists = "overwrite_terragrunt"
  contents = <<EOF
provider "aws" {
  region = "ap-southeast-2"
}
EOF
}

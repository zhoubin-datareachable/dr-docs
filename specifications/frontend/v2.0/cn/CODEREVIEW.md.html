描述

该文件是前端代码 review 规则，主要包含了如何进行组员之间的代码 reveiew。

为什么做 code review

code review 是在软件开发的过程中进行自检和同级评审，其目的是找出在软件开发过程中出现的错误，保证软件开发的质量，并且提高开发者的水平。

code review 标准

  • 是否存在 Eslint 报错
    在项目中，我们已经配置了 eslint 进行基本的错误排除，所以在 code review 的过程中第一个要检查的就是是否存在 eslint 的 error 或者 warning

  • css 的命名是否符合规范
    查看项目中 class 和 id 的命名是否符合规范，从而确保别的开发人员可以快速了解别人的项目。

  • 开发者是否滥用 css 中的 id,开发者是否选择了合适的 css 选择器
    查看项目中是否过多的使用了 id，某些 id 是否必须存在，是否可被 class 替换，是否可根据已有的 class 使用子代选择来替换过量的 class 等。

  • 开发者是否使用了正确的 HTML 标签
    查看项目中的 HTML 标签是否符合场景，例如，在给一段长文字添加标签的时候，是否使用了 p 标签，如果使用 div 标签，则无法通过 code review。

  • 开发者是否对代码添加了注释
    查看项目中的注释是否符合项目的注释规范，并且确保注释完善。

  • 开发者是否正确的给定量，变量，函数和类进行了命名
    查看项目中定量，变量，函数和类的命名是否符合项目命名规范。

  • 查看项目中的组建是否具有可读性
    查看项目中的组建是否具有可读性意味着组建是否过于复杂，是否可以进行拆分等。

  • 查看项目的代码是否存在重复代码
    查看项目中的代码是否存在重复的代码，确保重复的代码可以进行提取。

merge 分支与 tag 规范

  • 将要 merge 到 dev 分支的代码会在通过 code review 以后进行 merge, 这时需要对当前 dev 或者 test 分钟打入对应版本的 tag, tag 的命名规范主版本号与次版本号与版本计划发布表版本保持一致, 修正版本通过自然数增加, 即每提交一次代码, 修正版本号自动+1

版本命名规范

描述

该文档主要包含了如何对项目的版本进行命名, 以及命名的规范. 该命名规范主要使用在 git 中进行 ci/cd 的时候给 docker 打上 tag 的时候进行版本命名, 所有的版本命名应该以"v 主版本号.次版本号.修正版本号-附属标记"的方式进行命名, 例如: v1.0.1-dev

主版本号

  1. 项目/产品架构发生改变，主版本号+1
  2. 项目/产品主体结构件进行重大修改，主版本号+1
  3. 项目/产品主体构件接口协议发生更改，主版本号+1
  4. 此版本号变更由主需求方决定，所有模块统一

次版本号

  1. 新项目/产品立项，次版本号为 0
  2. 根据客户需求改进现有功能需求或性能，对现有功能模块做大的修改，但不增加新的功能，次版本号+1
  3. 产品增加新的功能，而产品的主体构件未做重大修改，次版本号+1
  4. 主版本号变更时，次版本号重置为 0
  5. 此版本号变更由产品部门决定，所有模块统一

修正版本号

  1. 测试过程中，每进行一次修改，修正版本号+1
  2. 次版本号变更时，修正版本号重置为 0
  3. 此版本号变更由各开发部门自行决定，模块间无需统一，部门间无需统一

附属标记

  1. dev：开发环境标识 ，为内部开发版，该版本 bug 较多，通常在新版本开发时使用
  2. test：测试环境标识，为内部展示版，该版本无影响功能流程的 bug，运行相对稳定，通常供内部展示所用

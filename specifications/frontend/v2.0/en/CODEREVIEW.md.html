Front End Code Review Rules

Description

This file mainly include the code review rules, it's about how to do the code review between different team members.

Why Code Review

Code review is a self-inspection and peer review in the process of software development. Its purpose is to find errors that occur in the process of software development, ensure the quality of software development, and improve the level of developers.

Code review standard

  • Is there an Eslint error
    In the project, we have configured eslint for basic error elimination, so the first thing to check in the code review process is whether there is an eslint error or warning

  • Whether the naming of css meet the requirements of the specification
    Check whether the naming of class and id in the project conform to the specification, so as to ensure that other developers can quickly understand other people's projects.

  • Whether the developer abuses the id in css, whether the developer chooses the appropriate css selector
    Check whether there are too many ids used in the project, whether certain ids must exist, whether they can be replaced by classes, whether they can use child selection to replace excessive classes based on existing classes, etc.

  • Whether the developer using the correct HTML tags
    Check whether the developer using the correct HTML tags, for example, in a paragraph, the developer is using div to wrapper the content, in this case, it will not pass the code review.

  • Does the developer complete the comments for the code
    Check whether the developer write corrent comments according to the project comment rules and make sure that the comments are sufficiently complete .

  • Does the developer give the correct name to the function, const, let and class
    Check whether the developer give correct name to the function, const, let and class according to the project naming rules.

  • Check whether the components in the project are readable
    Checking whether the components in the project are readable means whether the components are too complex, whether they can be split, etc.

  • Check whether there is duplicate code in the code of the project
    Check whether there are duplicate codes in the code in the project, and ensure that the duplicate codes can be extracted.

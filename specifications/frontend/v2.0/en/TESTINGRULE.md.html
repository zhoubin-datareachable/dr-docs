Front End Testing Rules

Description

This file mainly include the how to write testing in front end part, it's about where to put your testing and how to name your testing file.
For example: To write a testing file for the Header.tsx component, please create a new test file called Header.test.tsx under the _test_/component folder.

ProjectSample/ .................... Project Folder
├─ src/ ........................... Project Root Folder
│  └─ _test_ ...................... Project Testing Folder
│  │   ├─ Components/ ............. Project Public Components Testing Folder
└─ └─  └─ Header.test.tsx ......... Header Components Testing File

Naming your testing file

  • Testing tsx file The name of the tsx testing file should be start with the same name with your test file, and connect with test, end with tsx. for example. the Header.tsx component testing file should be named as Header.test.tsx.

  • Testing ts file The name of the ts testing file have the same naming rule with tsx testing file, but end with ts. For example, the configData.ts testing file will be named as configData.test.ts.

Which file need to write testing?

Due to the limitation of development progress, the BDD testing method could be the best choice. All the Front end developer need to write BDD or unit testing under the following circumstance.

  1. If you wirte the code according to the useCase, you must write testing.
  2. If the components you write involve business logics, you need to write testing.
  3. If the components you write involve user interaction, you need to write testing.
  4. If the function you write involve state change, you need to write testing.
  5. If the you write a class, you need to write testing.

What kind of testing is qualified？

Every testing have it's own testing coverage. After your write a testing file, you can simply run npm run coverage to see the coverage repot. The 100% coverage is the testing target. But, in BDD testing method, it's really hard to achieve this point. So we reduce the number to 80%.

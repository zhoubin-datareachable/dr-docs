README.md文件的书写规范（适用于所有项目）

⚠️ 已经废弃

本文档中规定的完整的README书写方式由于在实操中过于冗余，已经废弃。简化版本在此提供。

描述

本代码仓库定义了在数及公司书写项目中README.md文档的基本规范。本规范适用于所有应用程序开发项目和IaC项目。

规范列表

  • 项目标题（必须）

    项目标题应放在第一行，使用Markdown第二级标题样式：

    可选项：项目标题下方可以放置一个居中的图片用以描述此项目，也可以包含相关的GitHub的徽章 Shields。

  • 项目描述（必须）

    “项目描述”字样使用Markdown第二级标题样式。

    项目描述的内容使用Markdown的正常字体样式。

  • 项目目录结构（必须）

    “项目结构”字样使用Markdown第二级标题样式。

    项目目录结构 —— 子标题（必须）

    在项目结构里的子标题（子项）使用Markdown第三级标题样式。可以有如下一些子标题：

    1. 项目目录结构总览（必须）\

         项目源码的目录结构树状图总览。统一使用VSCode的扩展：[directory-tree generator](https://marketplace.visualstudio.com/items?itemName=yarimit.directory-configuration-generator)来自动生成（也可以手写）。

    2. 根目录描述（可选）\

         给出项目根目录下的每个文件和文件夹的详细描述。

    3. 源代码文件夹描述（可选）\

         给出源代码文件夹下面的每个文件和文件夹的详细描述。

    4. 其他文件夹描述（可选）
      有必要的情况下，可选的对其他文件夹内部的每个文件和子文件夹加以详细描述，比如，配置文件夹，测试文件夹等。

  • 使用方法（必须）

    使用方法小节包括了项目如何运行以及其他使用方法相关的内容。

    在描述使用方法之前，必须明确描述本地运行环境的要求，以及配置运行环境的方法，比如，必须的依赖库，最低的版本号等。这一项为必须写出的。

    针对不同的项目，使用方法小节所包含的具体内容会有不同，但是，必须保证用合适的层级来组织内容，同时给出相关的示例代码（此代码尽量保证可以直接拷贝后运行）。

  • 变更记录（可选）

    项目的变更记录不应该出现在首页的README.md文件中。变更记录文件应该被命名为CHANGELOG.md，并作为单独的文件与README.md文件并列存放。

    使用自动生成变更记录的工具（必须）

    如果选择添加项目的变更记录文件，那么强制使用如下自动生成工具以统一格式（Auto ChangeLog）。该工具依赖于 git logs 记录，通过提取git日志来自动生成格式化的变更记录文档。

    使用 Auto ChangeLog 工具应该首先通过npm install -g auto-changelog进行全局安装（注：该工具依赖于npm）。之后，使用如下命令来自动生成CHANGELOG.md文件。

    auto-changelog --hide-credit --unreleased --output CHANGELOG.md

    此工具主要用来记录GitHub上正式发布的版本信息。每次通过GitHub发布版本时，应该详细的描述改版本的相关信息。

    上述命令也会生成所有没有进行发布的git提交记录。

    当使用上述工具时，应该注意git的提交记录至关重要。因此，每一个提交所附带的信息应该统一格式，以便于阅读。请参阅：Git使用规范。

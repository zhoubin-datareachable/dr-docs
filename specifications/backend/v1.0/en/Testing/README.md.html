Testing Specifications

NOTICE: Writing test is the least important task for application developers. If you don't have time writing tests, leave it empty is ok.

Use jest as the standard testing framework for NodeJS projects. Jest already includes IstanbulJS libraries to provide code coverage, so there's no need to install other tools (e.g. nyc). Refer to configurations for jest configs.

Unit Testing Requirements

Unit testing aims at testing each module in the project in an isolated manner.

In unit testing, an emphemeral HTTP server might be started for certain tests.

The following components in a backend project should be tested independently:

  • Service

    Services define all low-level data manipulation logics, so they need to be tested. When connecting to databases, use either a real database connection or a mocked in-memory database (e.g. @shelf/jest-mongodb). We recommend to use a real testing database.

    Note: Insert testing data needed for a set of tests before the running the tests and clean up the testing data in the database once testing is completed.

  • Controller

    Controllers define all business logics to process incoming requests, so they need to be tested. When calling data services, use a mocked module to simulate the services, so that no database connections are needed. Unit tests on controllers should focus ONLY on the business logics inside controllers, assuming all data services are bug-free.

  • Utility

    Utilities define all the libraries needed by the project, so they need to be tested.

Integration Testing Requirements

Integration testing aims at testing a certain project feature as a whole.

In integration testing, an HTTP server running on localhost should be started and gracefully shutdown after the testing.

In integration testing, when database connections are required, connect to a real testing database.

Note: Insert testing data needed for a set of tests before the running the tests and clean up the testing data in the database once testing is completed.

Useful Tools

supertest package is a great tool to test HTTP Restful APIs.

Code Coverage

We consider 80% as the passing line for code coverage. This does NOT mean the overall code coverage should reach this line.

For unit testing, each tested unit should reach this line.

For integration testing, the routers, middlewares, controllers, services should reach this line. Utilities don't need to reach this line.

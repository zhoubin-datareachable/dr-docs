Naming Conventions

Naming conventions are important to keep the codebase consistent and maintainable. Several naming conventions are enforced and checked by the ESLint rules, including:

  • File name style
  • Naming style in the TypeScript source code

Naming Style for Folders and Files

Folders and files should be named in snake_case style, e.g. ctr_sample_controller.ts, rte_sample_router.ts, sample_view.ejs, sample_style.css. Other naming styles are NOT allowed.

ESLint checks on the folder/file naming styles. Ill-styled folder and file names are reported as ERROR.

Naming Style in Source Code

Names in the TypeScript language can be categorised as follows:

  1. variable - matches any var / let / const variable name.
  • Allowed modifiers: const, destructured, global, exported, unused.
  • Allowed types: boolean, string, number, function, array.
  1. function - matches any named function declaration or named function expression.
  • Allowed modifiers: global, exported, unused.
  • Allowed types: none.
  1. parameter - matches any function parameter. Does not match parameter properties.
  • Allowed modifiers: destructured, unused.
  • Allowed types: boolean, string, number, function, array.
  1. classProperty - matches any class property. Does not match properties that have direct function expression or arrow function expression values.
  • Allowed modifiers: abstract, private, protected, public, readonly, requiresQuotes, static.
  • Allowed types: boolean, string, number, function, array.
  1. objectLiteralProperty - matches any object literal property. Does not match properties that have direct function expression or arrow function expression values.
  • Allowed modifiers: public, requiresQuotes.
  • Allowed types: boolean, string, number, function, array.
  1. typeProperty - matches any object type property. Does not match properties that have direct function expression or arrow function expression values.
  • Allowed modifiers: public, readonly, requiresQuotes.
  • Allowed types: boolean, string, number, function, array.
  1. parameterProperty - matches any parameter property.
  • Allowed modifiers: private, protected, public, readonly.
  • Allowed types: boolean, string, number, function, array.
  1. classMethod - matches any class method. Also matches properties that have direct function expression or arrow function expression values. Does not match accessors.
  • Allowed modifiers: abstract, private, protected, public, requiresQuotes, static.
  • Allowed types: none.
  1. objectLiteralMethod - matches any object literal method. Also matches properties that have direct function expression or arrow function expression values. Does not match accessors.
  • Allowed modifiers: public, requiresQuotes.
  • Allowed types: none.
  1. typeMethod - matches any object type method. Also matches properties that have direct function expression or arrow function expression values. Does not match accessors.
  • Allowed modifiers: public, requiresQuotes.
  • Allowed types: none.
  1. accessor - matches any accessor.
  • Allowed modifiers: abstract, private, protected, public, requiresQuotes, static.
  • Allowed types: boolean, string, number, function, array.
  1. enumMember - matches any enum member.
  • Allowed modifiers: requiresQuotes.
  • Allowed types: none.
  1. class - matches any class declaration.
  • Allowed modifiers: abstract, exported, unused.
  • Allowed types: none.
  1. interface - matches any interface declaration.
  • Allowed modifiers: exported, unused.
  • Allowed types: none.
  1. typeAlias - matches any type alias declaration.
  • Allowed modifiers: exported, unused.
  • Allowed types: none.
  1. enum - matches any enum declaration.
  • Allowed modifiers: exported, unused.
  • Allowed types: none.
  1. typeParameter - matches any generic type parameter declaration.
  • Allowed modifiers: unused.
  • Allowed types: none.

Summary of allowed formats in the source code:

  • strictCamelCase - standard camelCase format - no underscores are allowed between characters, but consecutive capitals are not allowed (i.e. myId is valid, but myID is not).
  • StrictPascalCase - same as strictCamelCase, except the first character must be upper-case.
  • snake_case - standard snake_case format - all characters must be lower-case, and underscores are allowed.
  • UPPER_CASE - same as snake_case, except all characters must be upper-case.

We adopt the following naming conventions for the different categorised elements in the source code:

Format Allow Leading Underscore? Allow Trailing Underscore? Elements
strictCamelCase ❌ ❌ function, classMethod, typeMethod, accessor
strictCamelCase ❌ ❌ variable with "function" type, e.g. arrow functions assigned to a variable
UPPER_CASE, snake_case ✅ ✅ variable with "const" modifier
snake_case ✅ ❌ variable, parameter, classProperty, parameterProperty, typeProperty
StrictPascalCase ❌ ❌ class, interface, typeAlias, enum, typeParameter
UPPER_CASE, StrictPascalCase ❌ ❌ enumMember

objectLiteralProperty and objectLiteralMethod have been excluded from the above rules.
